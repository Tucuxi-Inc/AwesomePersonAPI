# PCP-MC13: Structural Problem Reframing

**Type:** Meta-Cognitive Pattern (Tier 3)
**Function:** Solving problems through structural reorganization rather than direct approaches
**Domain Independence:** Universal across organizational and system design contexts
**Orchestration Role:** Reframes problems to enable solutions unavailable through direct approaches

## Pattern Definition

### Trigger Condition
Situations where:
- Direct compliance or solution approaches are blocked or costly
- Organizational complexity creates barriers
- Multiple simple structures might work better than one complex one
- Corporate or process structure could be leveraged strategically
- Current approach has become unwieldy

### Core Procedure

1. **Structural Analysis**
   - What structures (corporate, process, organizational) are involved?
   - What constraints do current structures impose?
   - What alternative structures exist?

2. **Problem Reframing**
   - Can this problem be solved by restructuring rather than direct action?
   - What would the problem look like with different structural assumptions?
   - Are we solving the right problem at the right level?

3. **Structural Solution Generation**
   - What corporate entities could be used differently?
   - What process architectures could simplify this?
   - What organizational structures would make this easier?

4. **Complexity Optimization**
   - Is combined simplicity actually creating hidden complexity?
   - Would parallel simple structures outperform integrated complex ones?
   - What's the true total complexity cost?

5. **Implementation Path**
   - What structural changes are feasible?
   - What approvals/transitions are required?
   - What are the second-order effects of restructuring?

### Expert Heuristic

> "When you can't solve a problem within the current structure, change the structure. Sometimes two simple agreements beat one complex one; sometimes a different entity solves a compliance problem."

## Decision Criteria

**Criterion 1: Structural Solution Viability**
- IF direct approach is blocked, THEN explore structural alternatives
- IF structure change is easier than direct compliance, THEN restructure
- Rationale: Work around obstacles rather than through them when efficient

**Criterion 2: Complexity Optimization**
- IF combined structure creates hidden complexity, THEN decompose
- IF parallel simple structures reduce total burden, THEN parallelize
- Rationale: Apparent simplicity can mask actual complexity

**Criterion 3: Calibration Awareness**
- IF recently handled extreme complexity, THEN explicitly recalibrate
- IF current problem seems trivial, THEN verify calibration
- Rationale: Experience with extremes distorts normal perception

**Criterion 4: Transition Cost Assessment**
- IF structural change cost exceeds ongoing benefit, THEN work within current structure
- IF structural change enables ongoing efficiency, THEN invest in transition
- Rationale: Structure changes are investments, not just costs

## Contrast with Naive Approaches

**Naive Approach**: Try harder to solve problems within existing structures.
**Expert Approach**: Ask whether changing the structure would make the problem easier.

**Naive Approach**: Combine everything into one agreement/process for "simplicity."
**Expert Approach**: Evaluate whether parallel simple structures reduce total complexity.

**Naive Approach**: Accept structural constraints as fixed.
**Expert Approach**: Treat structures as design choices that can be modified.

**Naive Approach**: Use the same complexity calibration across all situations.
**Expert Approach**: Explicitly recalibrate after handling extreme situations.
